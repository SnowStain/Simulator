# 文档维护工作流

## 目标

这个仓库的功能跨度很大，既有 3D 渲染，又有规则仿真、地图处理、互动组件、自瞄、弹道、编辑器。如果功能改了但文档没跟上，后面继续改的人就会反复踩坑。所以文档维护必须被当成正式开发流程的一部分，而不是“有空再补”。

## 统一规则

每次功能更新，至少检查下面四件事：

1. `README.md` 是否需要更新
2. `docs/project-log.md` 是否已经记录
3. 对应算法文档是否已经同步
4. 如果新增入口、调试开关、编辑器能力，教学文档是否需要补充

## 什么时候必须更新 README

出现下面任一情况，就必须改 `README.md`：

- 启动方式变化
- 构建方式变化
- 地图目录结构变化
- 用户可见功能入口变化
- 键位、调试视图、主界面流程变化

## 什么时候必须更新算法文档

出现下面任一情况，就必须更新 `docs/algorithms/`：

- 自瞄目标建模变化
- 观测滤波器变化
- 提前量求解变化
- 弹丸碰撞或反弹变化
- 地图缓存格式变化
- 地形碰撞或地面贴合变化
- 能量机关、前哨站、基地等互动逻辑变化

## 什么时候必须更新架构文档

出现下面任一情况，就必须更新 `docs/architecture/`：

- 模块边界变化
- 工程目录职责变化
- 引入新的跨层消息或服务
- UI、渲染、规则、资源系统之间的调用方向变化

## 什么时候必须更新教学文档

出现下面任一情况，就必须更新 `docs/tutorials/csharp-beginner-project-guide.md`：

- 新增一个对初学者不直观的核心系统
- 主要入口文件变化
- 核心调试方式变化
- 代码阅读顺序发生明显变化
- 新增容易踩坑的工程约束

## 最小提交流程

推荐把一次开发拆成下面顺序：

1. 先改代码，把功能做通
2. 立刻补最小中文注释，至少把关键状态、单位、边界条件写清楚
3. 更新对应技术文档
4. 在 `docs/project-log.md` 追加当日记录
5. 如果入口或使用方式变了，再更新 `README.md`

## 不允许的文档维护方式

- 只改代码，不记日志
- 只记日志，不解释算法变化
- 只写“优化了”“修复了”，不说明影响范围
- 新增一个入口文件，但不告诉后来的人应该从哪里读起
- 把实现重构成新链路，却不说明旧链路是否还保留

## 推荐写法

写文档时尽量回答下面几个问题：

- 入口文件在哪
- 输入是什么
- 中间状态是什么
- 输出是什么
- 单位是什么
- 哪些是规则层决定的，哪些是渲染层决定的
- 如果出问题，优先看什么日志、什么类、什么方法

## 针对本仓库的强制同步点

下面这些模块每次改动，都要同步写文档：

- `TerrainMotionService`
- `SimulationCombatMath`
- `Simulator3dForm` 及其各个 partial
- `RuntimeGridData`
- `TerrainCacheCollisionSurface`
- `Simulator.LoadLargeTerrain`
- 地图目录下的 `json / glb / lz4 cache` 组织方式

## 提交前检查清单

- 我是否已经在 `docs/project-log.md` 写了今天的改动
- 我是否已经把受影响的算法文档同步更新
- 我是否已经说明新增链路是否替换旧链路
- 我是否已经写清楚 world-space、米、world unit 之间的换算
- 我是否已经说明新的调试开关或键位

如果这五项里有任何一项答案是否定的，这次更新就还不完整。
