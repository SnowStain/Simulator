# 给 C# 初学者的完整项目教学

## 1. 这份文档是给谁写的

这份文档是写给“会一点编程，但对 C#、.NET、WinForms、OpenTK、仿真项目、地图处理、自瞄链路都还不熟”的人。它不是只解释某一个函数，也不是只告诉你怎么运行项目，而是想把一整套“怎么看、怎么改、为什么这样分层、每一类代码分别负责什么”的思路讲清楚。

你可以把它理解成一份项目入门课。目标不是让你背代码，而是让你看到一个大仓库时不再发怵，知道从哪里下手，知道一个现象通常会落到哪一层，知道哪些变量是世界坐标、哪些变量是米、哪些状态属于规则层、哪些状态只属于渲染层。

如果你后面要继续改地图、组合体、互动组件、碰撞、自瞄、弹道、吊射、编辑器或者 GPU 调试显示，这份文档应该能给你一条稳定的阅读路径。

## 2. 先建立一个正确的心智模型

很多初学者第一次看这类项目，容易把整个仓库当成“一大堆混在一起的代码”。这样看是很痛苦的，因为你会不断遇到这些问题：

- 为什么一个功能同时改了 `Simulator.Core` 和 `Simulator.ThreeD`
- 为什么地图明明是资源文件，结果又牵扯到运行时网格、碰撞面、组合体、互动组件
- 为什么自瞄看起来像图形问题，最后却落到了规则层和数学层
- 为什么 `F3` 的显示和真实碰撞、真实发弹、自瞄锁定，不能各用一套独立逻辑

解决这些困惑的关键，是先把项目看成三层。

第一层是“规则层”。这一层负责回答“世界到底发生了什么”。谁在移动、谁被命中、谁在旋转、某个装甲板此刻世界坐标是多少、某个圆盘是不是已经激活、弹丸当前在哪里、下一帧会飞到哪里，这些都属于规则层的问题。规则层最核心的项目是 `src/Simulator.Core/`，但也有一部分运行时服务长在 `src/Simulator.ThreeD/`，例如结合地形和控制进行步进的 `TerrainMotionService`。

第二层是“渲染层”。这一层负责回答“如何把规则层的结果画出来”。它不应该创造新的规则事实，只应该消费规则状态。比如一个前哨站正在旋转，这个“正在旋转”应该由规则层或统一的运行时位姿链路给出，渲染层只负责把它画成旋转的样子。`src/Simulator.ThreeD/` 里绝大部分 `Simulator3dForm.*.cs` partial 文件都属于这一层。

第三层是“工具与编辑层”。它负责地图资源导入、地图编辑、组合体摆放、互动组件测试、辅助脚本启动等。比如 `src/Simulator.LoadLargeTerrain/`、`src/Simulator.Editors/`、根目录的 `run_viewer.py`、`open_csharp_project.py` 都属于这个范围。

你后面读代码时，只要先问自己一句：“我看到的这个问题，是世界状态错了，还是只是显示错了，还是资源导入错了？”排查范围会马上缩小很多。

## 3. 仓库目录怎么读

仓库根目录里最重要的不是文件数量，而是几个项目目录的职责边界。

### 3.1 `src/Simulator.Core`

这是规则和数学的核心层。你可以把它理解成“只关心世界如何运转，不关心窗口怎么画”的地方。这里通常会放：

- 世界状态数据结构
- 实体、弹丸、装甲板等模型
- 比赛规则
- 自瞄数学、弹道解算
- 命中判定

这一层最重要的好处是“可复用”。同样一套规则数学，不应该只能给一个界面使用。只要层次做对了，正式对局、测试器、编辑器里的预览、调试 HUD 都应该能共用同一套规则结果。

### 3.2 `src/Simulator.ThreeD`

这是当前正式 3D 对局和观察界面的主入口。虽然用户经常直接把它叫成“3D 程序”，但从代码组织上看，它并不只是一个窗口，它同时承担：

- 3D 场景显示
- GPU / CPU 渲染路径切换
- HUD
- 输入处理
- 视角控制
- 一部分实时运动服务和地图运行时桥接

这里最需要初学者理解的点，是 `Simulator3dForm` 不是一个小类，而是被拆成很多 partial 文件。也就是说，编译器最终看到的是“一个类”，但为了便于维护，它被分散在多个文件中。例如：

- `Simulator3dForm.cs`
- `Simulator3dForm.GpuRenderer.cs`
- `Simulator3dForm.AppearanceModel.cs`
- `Simulator3dForm.LiveControl.cs`
- `Simulator3dForm.FineTerrainActors.cs`

这是大项目里很常见的手法。你不能只打开一个主文件就以为看完整个类了。

### 3.3 `src/Simulator.LoadLargeTerrain`

这是新地图导入、精细地形、组合体处理、地图侧工具链的重要工程。用户现在要求新地图链路尽量向这个项目的处理方式靠拢，所以它既是功能模块，也是“参考实现”。如果你要理解：

- `glb` 如何进入运行时
- `lz4` 缓存怎么组织
- 组合体怎么做锚点和坐标系
- 地图编辑器为什么要和正式地图显示对齐

这个项目必须读。

### 3.4 `src/Simulator.Editors`

这里主要放编辑器共享控件和公共逻辑。它的价值在于减少不同编辑器之间的重复代码。如果以后要继续内聚地图编辑器、单位测试器、组合体编辑能力，这里往往是合适的落点。

### 3.5 `maps`

这里不是简单的贴图目录，而是运行时地图资产的根。里面会同时出现：

- 地图配置 JSON
- 场地说明
- `glb` 模型
- `lz4` 缓存
- 互动组件注释

初学者常见误区是“把地图当成一张图片”。这个项目里地图不是一张图，而是一整套资产和运行时处理链路。

### 3.6 `docs`

文档目录不是装饰。对于这种系统复杂度的工程，文档本身就是工程的一部分。你后面如果改了算法或入口但不改文档，几周后连你自己都会忘记之前为什么这么做。

## 4. 如果你对 C# 还不熟，先补这几个语法点

你不需要先学完整本 C# 语法书，但至少要熟悉下面这些项目里高频出现的东西。

### 4.1 类、字段、属性、方法

例如：

```csharp
internal sealed class TerrainMotionService
{
    private readonly RuleSet _rules;

    public void Step(SimulationWorldState world, RuntimeGridData? runtimeGrid, double deltaTimeSec)
    {
    }
}
```

这里有几个点：

- `class` 表示类
- `_rules` 是字段，通常表示这个对象长期持有的一份状态
- `Step(...)` 是方法，表示可以做一次世界推进
- `private` 表示只能在类内部访问
- `public` 表示外部也可以访问
- `internal` 表示同一个程序集内可见

### 4.2 `record struct`

项目里大量使用 `record struct`，例如 `ArmorPlateTarget`、`AutoAimSolution`。这类类型很适合“轻量、只读、用来传值”的数据结构。

你可以先粗略理解为：它比普通 class 更轻，适合表达“一个状态包”或者“一个计算结果”。如果你看到方法返回一个 `record struct`，大概率说明作者想把一组紧密相关的数据打包成一个稳定结果。

### 4.3 `List<T>` 和 `Dictionary<TKey, TValue>`

这两个是你读这个项目时最常见的容器。

- `List<T>`：有顺序的一组对象
- `Dictionary<TKey, TValue>`：通过 key 查 value 的哈希表

比如：

```csharp
private readonly Dictionary<string, AutoAimThirdOrderEkfPoseFilterState> _autoAimObservationEkf3Filters
    = new(StringComparer.OrdinalIgnoreCase);
```

这表示：每个 key 是一个字符串，value 是一份三阶 EKF 观测滤波状态。字符串比较使用大小写不敏感模式。

### 4.4 可空类型 `?`

例如：

```csharp
SimulationEntity? entity = _host.SelectedEntity;
```

这个问号表示“这里可能是 null”。你要养成习惯：看到 `?` 时，立刻想一想后面是不是做了空值判断。如果没有，很容易出空引用异常。

### 4.5 partial class

`Simulator3dForm` 被拆成多个 partial 文件。初学者最容易漏掉这一点，然后在一个文件里搜不到成员，就误以为不存在。实际上 partial 的意思是“同一个类被拆成多个文件写”。

所以你查一个字段或方法时，不要只看当前文件，要对整个目录做全文搜索。

### 4.6 命名约定

这个仓库里基本遵守下面的命名习惯：

- 私有字段：`_fieldName`
- 常量：`PascalCase` 或全局风格常量
- 方法：`PascalCase`
- 局部变量：`camelCase`

如果你自己继续写代码，跟着现有风格走，后面维护成本会低很多。

## 5. 项目如何启动

根目录 `README.md` 已经列出常用入口，这里换成初学者更容易理解的说法。

### 5.1 正式 3D 对局

最常见入口是：

```powershell
dotnet run --project src/Simulator.ThreeD/Simulator.ThreeD.csproj -- --start-match
```

这表示让 .NET 直接运行 `Simulator.ThreeD` 这个工程，并附带命令行参数 `--start-match`。

### 5.2 打开 C# 工程

如果你更习惯先打开 IDE 工程，可以运行：

```powershell
.venv\Scripts\python.exe open_csharp_project.py
```

这本质上是一个辅助脚本，帮你快速定位和启动 C# 工程，而不是项目逻辑本体。

### 5.3 Python 地图工具

地图查看和编辑辅助脚本：

```powershell
.venv\Scripts\python.exe run_viewer.py
.venv\Scripts\python.exe py_client\terrain_editor.py
```

这些脚本的价值不是替代正式程序，而是提供更轻量的资源检查入口。很多地图和组合体位置问题，用这类脚本更快定位。

## 6. 从“打开程序”到“世界开始跑”到底发生了什么

这是初学者最该建立的链路意识。你以后遇到任何问题，都可以把它映射到这条总链路上的某个节点。

大致流程是：

1. 程序启动，读取配置、窗口、地图、资源缓存
2. 进入对局或预览状态
3. 每一帧先推进世界状态，再进行渲染
4. 世界推进过程中会更新实体、弹丸、设施、互动组件、自瞄观测、AI 决策
5. 渲染阶段读取世界状态，把地形、单位、设施、HUD、调试信息画出来

这里有一个非常重要的原则：**世界推进和绘制最好是“状态生产”和“状态消费”的关系，而不是绘制函数自己发明一套状态。**

为什么这个原则重要？因为一旦你让渲染层自己重新算一遍目标位置、再让自瞄层自己重新算一遍、再让命中判定自己再算一遍，三者迟早会漂移。用户看到的现象就是：

- 眼睛看到目标在 A
- 自瞄锁到 B
- 弹丸判定命中的是 C

这个仓库很多修改，其实都是在修这种“多个链路各算一份”的问题。

## 7. 世界状态：项目里真正的“事实”

### 7.1 `SimulationWorldState`

你可以把 `SimulationWorldState` 理解成“这一时刻的整个世界快照”。它通常会持有：

- 实体列表
- 弹丸列表
- 队伍状态
- 规则状态
- 当前时间
- 单位换算参数

为什么它重要？因为只要世界推进函数拿到它，就能根据上一帧状态推出下一帧状态。只要渲染函数拿到它，就能画出这一帧的画面。

### 7.2 `SimulationEntity`

这是项目里最核心的数据结构之一。机器人、前哨站、基地、能量机关等很多可参与规则和显示的对象，都会映射成某种 `SimulationEntity`。

初学者要注意：`SimulationEntity` 不是“纯视觉模型”，它同时承载规则和运行时需要的信息，例如：

- 世界位置
- 朝向
- 高度
- 底盘参数
- 轮组参数
- 装甲板相关参数
- 设施的旋转/平移参数

这意味着当你改一个实体的参数时，你可能同时改变了：

- 它怎么画
- 它怎么碰撞
- 它怎么被自瞄识别
- 它怎么被弹丸命中

所以改实体参数时一定要有全链路意识。

### 7.3 `SimulationProjectile`

弹丸对象会保存发射者、速度、位置、弹种、寿命等信息。它的意义不是一条线，而是一个会被逐帧推进、会做碰撞检测、会触发命中反馈和交互逻辑的真实对象。

很多人一开始会把弹道预测线和真实弹丸混为一谈。要记住：预测线是调试显示，真实弹丸才参与规则。

## 8. 时间步进：为什么所有系统都离不开 `deltaTime`

### 8.1 什么是时间步进

仿真项目和静态软件最大的不一样，是它不是“点一次按钮做一次计算”，而是“每一帧都要继续推进”。这时最重要的输入之一，就是时间增量 `deltaTimeSec`。

如果某个速度是 `v`，经过 `dt` 秒，位移通常就是：

```math
\Delta x = v \cdot dt
```

如果你完全不考虑 `dt`，程序在 30 帧和 120 帧时就会表现完全不同。

### 8.2 `TerrainMotionService.Step`

这个方法是你后面读运动、地形、导航、自瞄更新时必须反复回来的地方。它承担了很多实时逻辑推进的工作。

初学者要注意，名字叫 `TerrainMotionService` 不代表它只管地形。它现在已经承担：

- 地形接触和移动
- 底盘跨越
- 一部分导航逻辑
- 自瞄观测状态更新
- 跟运动强相关的控制推进

这类类在大项目里很常见：最初只负责一件事，后面因为时序关系密切，被越来越多地挂载职责。你读它时，不要只按名字理解，而要按“它在一帧里到底做了哪些事”来理解。

### 8.3 为什么有很多 `Math.Clamp`

你会在项目里反复看到：

```csharp
double dt = Math.Clamp(deltaTimeSec, 0.01, 0.2);
```

这是因为真实运行时不稳定：

- 程序卡顿时，一帧可能很长
- 程序切后台再回来时，时间差可能异常大
- 某些滤波或积分算法对过大的 `dt` 很敏感

所以工程里常常会夹一个上下限，避免单次更新把系统拉爆。

## 9. 地图系统：为什么这个项目的地图不是一张图片

### 9.1 旧地图和新地图是隔离的

这个仓库同时维护两套地图体系：

- `blankCanvas`：老的粗略地图或实验场
- `rmuc2026`：新的精细地图

用户已经明确要求新地图和老地图设施链路要剥离。你在代码里一定要保持这个意识：不要为了图省事，把老设施逻辑偷偷混到新地图里。

### 9.2 地图资产的组成

一张新地图在这个项目里通常不只是一个模型文件，而是一整组东西：

- 地形几何
- 表面材质或颜色信息
- 组合体
- 互动组件
- 配置 JSON
- 缓存文件，例如 `lz4`

也就是说，“地图加载”实际上是多步转换。

### 9.3 为什么要做缓存

如果每次启动都完整解析重地图，再逐块重新建立所有运行时结构，启动时间和帧内卡顿都会很明显。缓存的核心目的，是把代价高的预处理结果保存下来，下次优先复用。

在这个项目里，缓存不只是为了快，也是为了稳定。因为你一旦把导入流程规范化了，地图编辑器、正式对局、单位测试器就更容易共用一套规范后的 world-space 结果。

### 9.4 `RuntimeGridData`

`RuntimeGridData` 可以理解为“地图进入运行时后的可查询形式”。它不等于原始资源，而是一个更适合：

- 采样地形高度
- 查询碰撞
- 调试三角面
- 提供移动和地形贴合服务

的中间层对象。

这一步非常关键。你几乎不可能拿着原始 `glb` 直接高效完成所有规则计算，所以一定会有“资源格式 -> 运行时格式”的转换层。

## 10. 地形碰撞：从三角面到可查询碰撞面

### 10.1 为什么三角面很重要

用户已经反复强调，这张新地图不能再退回到粗栅格近似，而要尽量基于真实三角面来表达地形。原因很直接：如果地形细节被过度离散化，程序就会出现这些问题：

- 斜坡失真
- 台阶高度错
- 视觉上没障碍，碰撞上却被挡住
- 视觉上地面连续，运动上却会卡死

所以三角面不是“渲染细节”，而是碰撞正确性的基础。

### 10.2 `TerrainCacheCollisionSurface`

这是地形碰撞里最关键的类型之一。它把地形三角面组织成更适合查询的结构，常见职责包括：

- 记录三角面
- 建立空间分块
- 做高度采样
- 做射线或路径相关的碰撞判断
- 输出局部调试三角面

初学者要理解一个工程事实：**碰撞系统真正关心的不是模型看起来有多漂亮，而是查询是否快、结果是否稳定。**

所以同一份几何数据，常常会有：

- 一份适合渲染的表示
- 一份适合碰撞查询的表示

它们不一定长得一样，但必须表达同一块空间事实。

### 10.3 `F3` 为什么能看到碰撞面

`F3` 的调试不是凭空画几条线，而是从运行时碰撞结构里把局部相关三角面取出来，再画到画面上。这样做的好处是：当用户说“这里明明没东西，但车过不去”时，你能直接看到系统实际认为的碰撞面。

这也是为什么本轮修改把 `F3` 的碰撞体积和局部地形碰撞显示补到了 GPU 端。因为如果调试几何本身走的是另一套慢路径，你排查性能问题时会被调试显示本身干扰。

## 11. 机器人运动与地形贴合

### 11.1 用户最在意的不是“能动”，而是“不能穿模、不能卡死”

在这个项目里，运动系统不是简单的平面位移。用户反复提出的要求都指向同一个核心：机器人要真的“贴着地形”运动，而不是只是把 XY 改掉。

典型问题包括：

- 下台阶时卡边
- 上坡时底盘姿态不对
- 轮子悬空却还在驱动
- 视觉模型在地面上，碰撞体却卡在空中
- 明明前面空的，却被不存在的物体挡住

### 11.2 轮组、底盘和姿态不是一回事

初学者很容易把“实体位置”理解成一个点。但车辆至少有三种相关几何：

- 底盘中心
- 轮组接地点
- 碰撞体积

真正稳定的地形贴合，通常不是“先摆底盘再算轮子”，而是“先看轮子与地形的关系，再反推出合理底盘姿态”。这也是为什么用户一直强调四轮贴地、至少三轮贴地、剩余轮子绑定到底盘。

### 11.3 运动系统的工程目标

你继续改运动时，要同时满足几件事：

- 不穿模
- 不悬空卡死
- 斜坡和平缓地形要顺滑
- 台阶要按设定跨越能力处理
- 轮组视觉和碰撞不要分家

这些目标彼此是有张力的。碰撞过严会卡死，碰撞过松会穿模。工程上真正难的不是写一个“碰撞检测函数”，而是把这些目标平衡到一个用户能接受的状态。

### 11.4 为什么用户总让你看 `F3`

因为 `F3` 是运动问题的第一现场。只看最终画面，你很难判断是：

- 地图碰撞面错了
- 机器人碰撞体积错了
- 地形采样错了
- 姿态重建错了
- 还是控制逻辑本身错了

而 `F3` 正是把这些“系统眼中的真实几何”暴露出来。

## 12. 碰撞体积和视觉模型为什么不能各玩各的

### 12.1 视觉模型正常，不代表碰撞正常

用户经常会说“模型看着是对的，但车还是会卡住”。这说明视觉模型和碰撞体积没有保持一致。

很多工程里视觉模型和碰撞模型会分开，这是合理的，因为视觉模型太复杂，直接参与实时碰撞代价高。但“分开”不等于“脱节”。至少要满足：

- 同一坐标系
- 同一世界位姿
- 同一锚点语义
- 同一整体变换链路

### 12.2 `EntityCollisionModel`

这个类负责为实体解析出碰撞部分，例如：

- 车体
- 云台
- 轮子
- 后腿或爬坡辅助结构
- 设施结构底座和附加臂

你可以把它理解成“把复杂实体拆成若干规则可控的碰撞部件”。

初学者常见误区是：觉得碰撞模型越精细越好。实际上不完全对。碰撞模型的目标不是复制视觉细节，而是在计算成本和物理稳定性之间找到一个合适的中间层。太粗会挡错，太细会抖动、穿透、算不动。

### 12.3 为什么本轮要求把 `F3` 碰撞体积画到 GPU

因为碰撞体积本身已经是运行时的重要几何。如果 `F3` 时又临时走一套 CPU/GDI 叠加，不仅会额外拖帧，还会让调试显示与真实渲染链路分离。现在改成 GPU 后，至少在 GPU 模式里，碰撞箱的绘制更接近正式几何路径。

## 13. 弹丸链路：从发射到命中

### 13.1 发射不是“按键立刻命中”

这个项目里的弹丸是独立的运行时对象。发射时会创建弹丸，然后每帧推进位置，再做碰撞检测，再决定是否命中某个实体或互动组件。

这条链路至少包含：

- 发射初速度
- 发射方向
- 重力
- 可能的空气阻力或简化模型
- 与地形的碰撞
- 与装甲板、圆盘、互动组件的碰撞

### 13.2 为什么自瞄和弹丸是两回事

自瞄算出来的是“往哪里打更可能命中”，弹丸系统算的是“真实发出去以后会怎样飞”。如果你把两者混成一个函数，短期看起来省事，长期一定出错。

正确关系应该是：

1. 自瞄给出目标点和期望角度
2. 控制系统按这个结果调整云台
3. 触发发射后，弹丸系统用真实发射状态独立推进
4. 命中判定基于弹丸与目标碰撞

### 13.3 为什么命中反馈也要统一

能量机关、前哨站、基地顶部装甲板的灯条反馈，看起来像纯视觉效果，但其实它们应该挂在“真实命中结果”之后。如果你让渲染层自己猜“应该闪一下”，最后就会出现明明没命中却在闪、或者明明命中了但没反应。

## 14. 自瞄链路总览：从观测到开火

### 14.1 先统一目标建模

这个项目里，无论是车体装甲板、前哨站旋转板、基地顶部装甲板，还是能量机关圆盘，最终都尽量统一成 `ArmorPlateTarget` 这种可求解目标。

为什么要统一？因为后面的链路基本一样：

- 先观测
- 再滤波
- 再预测未来位置
- 再求解弹道
- 再决定是否自动扳机

如果每类目标都用完全不同的一套接口，你后面会越来越难维护。

### 14.2 观测目标和攻击目标可以不同

能量机关就是最典型的例子。用户要求用 `1 环` 做视觉观测，用 `10 环` 做真实攻击。也就是说：

- 观测点为了稳定
- 攻击点为了命中规则

这说明项目里的“看见什么”与“最后打什么”不一定是同一个点，但它们必须由一条清楚的映射链路连接起来。

### 14.3 旧链路和新链路为什么要并行保留

用户明确要求三阶 EKF 新建链路，不要破坏现在的链路。这是一个很正确的工程要求。因为观测滤波属于高风险修改，一旦直接替换旧链路：

- 旧功能可能被连带破坏
- 新问题很难快速回退
- 你无法做 A/B 对比

所以当前实现里，旧的常速度 Kalman 和新的三阶 EKF 是并行存在的。这样做的好处是：

- 可以逐步验证
- 可以快速定位问题是新链路还是旧业务逻辑
- 可以在未完全稳定前保留回退空间

## 15. 三阶 EKF 到底解决什么问题

### 15.1 旧的常速度模型有什么局限

常速度 Kalman 的核心假设是“目标短时间内大致匀速”。这对缓慢移动目标常常够用，但对下面这些情况就不够：

- 小陀螺快速转板
- 能量机关圆盘快速转动
- 前哨站旋转装甲板
- 目标短时加减速明显

因为如果模型里没有加速度项，系统会把所有弯曲变化都当成噪声，结果就是：

- 预测滞后
- 提前量偏少
- 目标快转离后才开火

### 15.2 三阶状态怎么理解

这里的“三阶”不是说用了三个滤波器，而是说状态里显式包含了位置、速度、加速度。

平面状态可以写成：

```math
\mathbf{x}_{xy} =
\begin{bmatrix}
x & y & v_x & v_y & a_x & a_y
\end{bmatrix}^T
```

高度状态可以写成：

```math
\mathbf{x}_{z} =
\begin{bmatrix}
z & v_z & a_z
\end{bmatrix}^T
```

这意味着系统不仅知道“目标现在在哪、朝哪里走”，还会维护“它现在正在怎样加速变化”。

### 15.3 为什么叫 EKF

因为平面观测不是直接用笛卡尔坐标，而是相对射手的：

- 距离 `range`
- 方位角 `bearing`

这个观测模型是非线性的，所以不能直接套最简单的线性 Kalman，需要用扩展卡尔曼滤波的思路在当前估计点做线性化。

### 15.4 对工程有什么直接收益

用更工程化的话说，三阶 EKF 的收益是：

- 短时丢目标后，预测更稳定
- 快速转板时，提前量不容易总是偏晚
- 观测噪声和真实机动更容易区分
- 最终弹道求解拿到的是更“像真实运动”的未来点

这并不意味着新链路永远更强。滤波器好不好，取决于模型是否匹配、噪声参数是否合理、下游求解是不是也跟着一起升级。所以本项目这次不是只把滤波器换掉，而是同时让提前量求解支持速度和加速度。

## 16. 提前量为什么不能只算一次

### 16.1 因为飞行时间本身就依赖目标未来位置

如果目标静止，提前量很简单。但目标在动时，你要先猜“弹丸飞多久”，而飞多久又取决于你最终瞄向哪里；最终瞄向哪里，又取决于目标未来在哪。它本质上是一个相互耦合的问题。

### 16.2 迭代求解的直觉

可以这样理解：

1. 先按当前距离估一个飞行时间
2. 用这个时间去预测目标未来位置
3. 再针对那个未来位置反求新的射击角
4. 再重新估飞行时间
5. 循环几轮，直到结果稳定

这就是为什么自瞄里经常会看到迭代，而不是一个一步到位的闭式公式。

### 16.3 新链路里为什么要把加速度带进去

因为如果你已经用三阶 EKF 估出了加速度，但提前量还是只按：

```math
p = p_0 + vt
```

来算，那前面一半收益就丢了。现在新链路对普通目标使用：

```math
p = p_0 + vt + \frac{1}{2}at^2
```

这样至少在短时间预测上，模型前后一致。

## 17. 自动扳机：为什么不是“准星必须完美重合”才开火

用户要求英雄吊射和自动扳机的逻辑尽量实用，不要把判定门槛设得过于苛刻。这里的工程关键点是：自动扳机不是追求数学上的零误差，而是追求“在当前弹丸、当前目标尺寸、当前预测误差下，已经满足命中窗口”。

换句话说，自动扳机的条件应该更像：

- 预测命中点已经落进装甲板有效区域
- 或者准星落在一个允许误差窗内
- 同时云台状态和弹道结果满足发射要求

而不是：

- 准心必须和板中心完全一点不差

后者在真实系统里几乎永远太苛刻。

---

## 18. 英雄吊射模式怎么理解

用户对英雄吊射的要求比较特别，本质上是把它做成一个“高弹道、自动扳机、可以保留上一时刻目标运动估计”的专用模式。

从工程角度看，吊射模式和普通自瞄的区别主要在几个方面：

- 搜索目标范围和目标类型不同
- 允许更远距离锁定结构目标
- 弹道求解参数不同
- 自动扳机逻辑不同
- 界面上需要额外的命中偏差或命中预览显示

所以不要把吊射只理解成“pitch 再抬高一点”。它其实是自瞄链路的一个分支模式。

## 19. 组合体和互动组件：新地图设施为什么不能只加载互动体

这是最近用户反复指出的问题。要理解这个问题，先分清两个概念。

### 19.1 组合体

组合体是“一整组应该一起动的模型”。它可能有：

- body
- 支架
- 转轴
- 外框
- 连接件

### 19.2 互动组件

互动组件是组合体内部真正参与规则交互的部分，例如：

- 装甲板
- 圆环
- 灯条
- 可命中区域

### 19.3 为什么先加载组合体，再挂互动组件

因为互动组件通常不是独立漂在世界里的，它是“跟着组合体一起动”的。如果你只加载互动组件，不加载组合体本身的变换链路，就会出现：

- 位置偏移
- 锚点找不到
- 旋转轴对不上
- 自瞄锁点和肉眼所见不一致

这也是用户后来明确指出的问题：不是互动组件定义错了，而是你没有先让组合体本体走到正确世界位姿。

## 20. 坐标系、锚点和 world-space 规范化

### 20.1 为什么“看着差不多”不行

地图编辑器里位置正确，进入正式对局就偏移，这种问题几乎都和坐标系链路有关。初学者经常只改一个最终位置值，希望“看起来差不多”，但这类问题如果不追根到底，后面一定反复出现。

### 20.2 正确的问题拆法

遇到组合体偏移时，至少要检查四件事：

1. 原始资源坐标系是什么
2. 导入时做了什么轴变换
3. world-space 规范化时做了什么平移和缩放
4. 运行时是否又额外叠加了一次老链路位移

### 20.3 为什么要统一成 world-space

因为只有统一成世界空间后：

- 地图编辑器
- 单位测试器
- 正式对局
- 自瞄目标生成
- 命中判定

才能共享同一套位姿结果。否则每个系统都拿自己的局部坐标系做一遍变换，你很快就会失去可维护性。

## 21. 渲染层：CPU 路径、GPU 路径和 OpenTK

### 21.1 为什么一个窗体项目里会有 GPU 渲染

很多初学者会误以为 WinForms 就只能画 GDI。实际上这个项目当前是“窗体宿主 + GPU 渲染上下文”的混合模式。也就是说，窗口底层仍然是桌面应用框架，但 3D 主要绘制可以走 OpenGL / OpenTK 路线。

### 21.2 为什么还保留了一部分 GDI/HUD

因为文字、HUD、一些 2D 叠加内容，用 GDI 或 2D 覆盖层依然方便。关键不是“是否纯 GPU”，而是“同一份几何事实不要在 CPU 和 GPU 各算一遍不同版本”。

所以本轮对 `F3` 的要求是：

- 统计文字可以还是 HUD
- 但碰撞几何本身要走 GPU

### 21.3 `Simulator3dForm.GpuRenderer.cs`

如果你想读 GPU 渲染主链路，这个 partial 是非常重要的入口。它会负责：

- 建立 GPU 上下文
- 设置视口和矩阵
- 画地形底图和几何
- 画设施
- 画实体
- 画弹丸
- 刷新 GPU 批次

你可以把它理解成“把当前世界状态翻译成 GPU 指令”的主调度器。

### 21.4 为什么项目里既有即时绘制也有批处理

因为工程处于演进中。一些功能会直接调用立即模式绘制函数，一些则已经积累到动态批次里统一 flush。初学者看到这类混合状态不要慌，它不是“写错了”，而是系统在逐步优化过程中的过渡形态。

真正重要的是读清楚：

- 哪些函数只是把几何 append 到 buffer
- 哪些函数是真正把 buffer 提交到 GPU
- 哪些函数只在调试模式下执行

## 22. 为什么 `Simulator3dForm` 要拆成很多 partial

因为这个类已经承担了太多职责，如果全塞进一个文件，会完全不可读。初学者可以按下面方式理解这些 partial：

- 主文件：状态、主循环、HUD 汇总
- `GpuRenderer`：GPU 渲染实现
- `AppearanceModel`：机器人和设施外观模型拼装
- `LiveControl`：实时控制和交互
- `FineTerrainActors`：精细地形相关对象
- `Structures`：设施模型与结构相关渲染

读这种 partial 类时，一个非常实用的方法是：**不要按文件顺序读，要按问题顺序读。**

比如你现在追“为什么 `F3` 下碰撞箱不是 GPU 画的”，那你该读的是：

1. `Simulator3dForm.cs` 里 `F3` 调用入口
2. `DrawEntityCollisionBox`
3. `DrawPrismWireframe`
4. `Simulator3dForm.GpuRenderer.cs` 里的动态几何 append / flush

而不是从文件第一行开始顺读。

## 23. 地图编辑器和单位测试器为什么重要

很多项目把编辑器当成“附属工具”，但这个仓库不是这样。这里的编辑器和测试器实际上承担了两类很重要的职责：

- 让地图和组合体能够被摆正、保存、复用
- 让复杂互动体可以脱离正式对局单独验证

对于新地图设施来说，这尤其重要。因为你不可能每次都进完整比赛流程，只为了看一个前哨站顶部板是不是偏移了 20 厘米。

所以编辑器和测试器不只是辅助，它们是降低调试成本的核心基础设施。

## 24. 调试时应该优先看哪些键位和日志

用户已经长期依赖几个调试入口，这些入口你以后改动时一定要小心。

### 24.1 `F3`

主要看碰撞、姿态、轮子离地、局部地形碰撞调试。

### 24.2 `F5`

用于显示按键说明和一些用户侧提示。因为用户已经要求这里承担操作说明，所以如果你新增了局内编辑或视图切换，应该同步更新这里。

### 24.3 `F8`

是自瞄与视觉解算的重要调试入口。用户很多时候不是先说“某个函数错了”，而是说“F8 看到的东西不对”。这时你就要理解，F8 是整条自瞄链路可视化的外壳。

### 24.4 `F9`

局内编辑器相关能力。如果这个功能没有响应，通常要排查：

- 键位入口有没有接到
- 当前状态是否允许进入编辑模式
- 选中对象逻辑是否成立
- 坐标变换是否按组合体坐标系执行

### 24.5 日志

工作区里已经积累了不少日志文件，例如：

- `render_perf.log`
- `motion_perf.log`
- `rule_perf.log`
- `terrain_movement_block.log`
- `gc_control.log`

读日志时不要只看“有没有报错”，更要看：

- 某一帧是不是突然耗时尖峰
- 是渲染、规则、运动还是垃圾回收在尖峰
- 某个现象出现前后，系统状态有没有明显变化

## 26. 初学者最容易踩的坑

### 26.1 把世界单位和米混了

这是最常见也最隐蔽的问题。项目里平面很多地方是 `world unit`，高度常常直接是米。只要你漏乘一次 `MetersPerWorldUnit`，结果就会“看起来差一点，但整条链路都不对”。

### 26.2 改了渲染位置，但没改规则位置

这样会导致：

- 肉眼位置对了
- 自瞄还是锁旧位置
- 碰撞还是旧位置

短期看起来你像是“修好了一个显示 bug”，实际上只是把真实 bug 藏起来。

### 26.3 只改组合体，不改互动组件

或者反过来只改互动组件，不改组合体。二者如果不一起遵循同一变换链路，最后一定偏移。

### 26.4 为了修一个现象，偷偷复用老链路

比如新地图设施没对齐，就把老地图设施坐标链路临时接回来。这种做法短期能出图，长期会让新旧系统继续纠缠。

### 26.5 不写日志和文档

这不是“流程要求”，而是避免未来反复返工的必要条件。这个项目的复杂度已经高到，如果你两周不写文档，后面再改几乎一定会重新踩自己之前踩过的坑。

## 27. 如果你要加一个新功能，推荐的落地顺序

假设你要新增一个可旋转、可被弹丸触发、可发光的互动设施，推荐顺序是：

1. 先定义资源层表示：模型、注释、互动组件命名
2. 再定义运行时组合体和互动组件数据结构
3. 建立 world-space 规范化结果
4. 先在编辑器或单位测试器里验证位置和旋转轴
5. 再接入正式对局的规则更新
6. 再接入渲染和自瞄
7. 最后补日志和技术文档

为什么这个顺序合理？因为它从“资源正确”开始，而不是一上来就冲进正式对局。正式对局是链路最复杂、变量最多的地方，不适合作为第一调试现场。

## 28. 如果你要查一个 bug，推荐的排查顺序

以“前哨站装甲板看见了但锁不到”为例，推荐按下面顺序查：

1. 组合体 world-space 位姿对不对
2. 互动组件是不是绑定在组合体下
3. 运行时目标生成有没有把这块板导出成 `ArmorPlateTarget`
4. 自瞄搜索阶段有没有过滤掉它
5. 观测和未来预测链路有没有把它当成结构目标处理
6. `F8` 调试里画出来的目标框是不是与肉眼一致

这就是典型的“从资源 -> 运行时目标 -> 搜索 -> 观测 -> 求解 -> 可视化”排查法。

## 30. 给初学者的最终阅读路线

如果你今天第一次接触这个项目，建议按下面顺序读和跑。

第一步，先读根目录 `README.md`，知道项目入口、目录和主要文档。

第二步，打开 `docs/README.md`，知道有哪些主题文档。

第三步，通读本文，先建立整体结构认识，不要急着看每个函数。

第四步，按你最关心的问题分支阅读：

- 地图问题：先看地图处理文档，再看 `Simulator.LoadLargeTerrain`
- 运动问题：先看地形与碰撞文档，再看 `TerrainMotionService`
- 自瞄问题：先看自瞄文档，再看 `SimulationCombatMath` 和 `TerrainMotionService`
- 渲染问题：先看 `Simulator3dForm.GpuRenderer.cs` 和相关 partial

第五步，带着一个很具体的问题去读代码，例如：

- 为什么这里会卡边
- 为什么这里会偏移
- 为什么这里锁定不到
- 为什么这里帧率掉了

不要一开始就问“这个项目整体是怎么工作的”，那样太大了。先用本文建立骨架，再用具体问题钻进去，效率最高。

## 31. 结语：你真正要学会的不是背代码，而是拆链路

这个项目里最有价值的能力，不是记住某个函数名，而是学会把一个现象拆成链路：

- 资源输入在哪
- 运行时状态在哪
- 规则更新在哪
- 渲染消费在哪
- 调试显示在哪

只要你能稳定地这样拆，一个再复杂的现象也会慢慢缩成几个可检查的点。

这也是这份文档真正想教给你的东西。代码会继续变，类会继续拆，渲染路径会继续重构，地图系统也会继续演进，但“按链路理解系统”的方法不会过时。

后续只要仓库继续更新，这份文档也应该跟着一起更新。因为对于这种复杂工程，教学文档不是额外产物，它本身就是项目可维护性的组成部分。

## 32. 一个完整的实战排查案例：为什么“看见模型了，但自瞄还是锁偏”

这一类问题在本项目里极常见，因为它同时跨越资源、规则、渲染和 HUD 四层。下面给一个完整排查示例，帮助你建立真正的工程思路。

假设现在现象是：用户看见前哨站旋转装甲板已经在正确位置旋转，但 `F8` 锁定点还是偏到下方，或者压根锁不到。很多初学者会立刻去改自瞄评分函数，觉得“应该是目标筛选错了”。但如果你按链路排查，顺序应该完全不同。

第一步，先确认这块板有没有被导出成运行时目标。也就是说，不要先看画面，要先看生成 `ArmorPlateTarget` 的代码链路。因为只要运行时目标都没建出来，后面所有搜索、观测、预测都无从谈起。

第二步，确认导出的目标点坐标和组合体 world-space 位姿是不是一致。这里最容易出错的是把互动组件局部坐标直接当成世界坐标使用，或者组合体 body 的位移矩阵和互动组件的位移矩阵没有复用同一套结果。用户前面多次指出“模型位置对，但互动体偏移”，本质上就是这一类错误。

第三步，确认渲染层画出来的位置是不是直接消费那份目标数据，还是偷偷重新算了一份“看起来差不多的位置”。如果 HUD 或 `F8` 自己再算一遍，就可能出现模型和锁点不一致。

第四步，确认观测目标和攻击目标的映射是不是正确。能量机关就是最典型的例子，观测环和攻击环不一样。如果你拿观测环的点直接当攻击点，自瞄看起来像“有目标”，但命中率会非常差。

第五步，确认观测滤波器是不是在使用正确的板。旧链路和新三阶 EKF 链路现在是并行存在的，如果 key 拼接不一致，或者 observation plate 和 aim plate 搞反了，就会出现“滤波器在跟踪一块板，发射却打另一块板”的现象。

第六步，确认提前量求解是不是把结构目标错误当成普通运动目标。像前哨站旋转板、基地顶部板、能量机关圆盘，这些都不应该只靠简单直线外推，而应尽量走未来规则几何重建。否则你看到的偏差常常不是滤波器错，而是未来位姿求解模型错。

第七步，再回来看评分、锁定保持和自动扳机。如果前六步都没问题，这一步才有意义。否则你很可能会在末端逻辑上补丁式修复，把本来应该在前端链路解决的问题掩盖掉。

这个案例的核心不是前哨站本身，而是教你一个方法：**先确认目标是否存在，再确认位姿是否一致，再确认观测是否正确，最后才去改策略和打分。**

## 33. C# 代码阅读速查表：看不懂时先用这张表定位

很多初学者不是数学看不懂，而是连代码形状都没认出来。下面给一个针对本仓库的速查表。

### 33.1 看到 `readonly`

说明这个字段或结构在初始化后不希望再被修改。通常用于表达“这份数据应该稳定”，例如配置、引用、轻量结果结构等。

### 33.2 看到 `sealed`

说明这个类不打算再被继承。对阅读者来说，它的直接意义是：你不用继续猜“会不会有一个子类偷偷改行为”。这在大型工程里能明显降低理解成本。

### 33.3 看到 `internal`

说明作者希望它只在当前程序集内使用。对初学者来说，这通常意味着“它是工程内部实现，不一定是公共 API”。

### 33.4 看到 `record struct`

先把它当成“轻量数据包”。读这类类型时优先关心字段含义和单位，不要先纠结高级语法细节。

### 33.5 看到 `TryXxx(...)`

这类函数通常遵循“尝试做某事，成功返回 true，失败返回 false”的风格。比如 `TryResolveObservedPlateAngularVelocityFromFilteredState(...)`。读这类函数时，第一眼先找：

- 失败条件是什么
- 成功时额外返回了什么 `out` 参数

### 33.6 看到 `ResolveXxx(...)`

通常表示“根据现有状态，解析或求出某个结果”。它不一定会失败，也不一定会改状态。很多几何、目标建模、显示相关函数都喜欢用这个前缀。

### 33.7 看到 `UpdateXxx(...)`

大概率说明函数会改动对象内部状态。比如滤波器、缓存、字典、定时器之类。你读它时要特别留意副作用。

### 33.8 看到 `DrawXxx(...)`

不要下意识以为它只画图。在这个项目里有些 `Draw` 函数会：

- 收集几何
- 追加 GPU batch
- 顺手更新局部缓存

所以读渲染函数时也要警惕它是否带有副作用。

### 33.9 看到 `out double value`

说明函数除了返回值外，还想额外把某个结果带出去。这种写法在性能敏感或“成功/失败 + 结果”风格函数里很常见。

### 33.10 看到 `Math.Clamp`

这通常意味着作者在给数值做边界保护。常见动机有：

- 防止时间步长过大
- 防止角度、概率、亮度越界
- 防止测量噪声太夸张
- 防止除零或极端输入把系统拉崩

### 33.11 看到 `StringComparison.OrdinalIgnoreCase`

说明字符串比较不区分大小写。对于 id、类型名、地图组件命名，这很常见。你自己写新代码时也尽量跟现有风格保持一致。

### 33.12 看到很多小函数互相调用

不要烦躁。这通常不是“写得太碎”，而是为了把：

- 坐标变换
- 条件判断
- 噪声参数
- 目标类型分支

拆成更容易替换和调试的片段。大工程里，一个 500 行大函数通常比 10 个 30 行函数更难维护。

## 34. 如何长期维护这个仓库，而不是只修眼前问题

真正成熟的工程实践，不是“把今天的 bug 修掉”，而是让后面的修改越来越不容易把系统搞乱。对这个仓库来说，长期维护最重要的是下面五件事。

第一，把“同一事实只算一遍”作为最高原则。组合体位姿、装甲板目标、圆盘目标、碰撞体积、自瞄锁点、命中判定，都要尽量消费同一份上游结果，而不是各算一份近似值。

第二，把单位写清楚。world-space、米、角度、弧度、world unit，这些只要混掉一个，整个系统就会产生诡异的偏差。初学者最该养成的习惯，是在读公式和函数签名时先问“单位是什么”。

第三，坚持把新链路并行接入，而不是直接替换旧链路。三阶 EKF 这次就是正确例子。高风险系统先并行、再验证、再决定是否切主路径，远比一步替换安全。

第四，尽量先在编辑器或测试器里验证资源和位姿问题，再进入正式对局。正式对局是变量最多的地方，不适合作为第一排查现场。

第五，每次改完都写文档和项目日志。不是因为流程漂亮，而是因为这个仓库的复杂度已经决定了“记忆”不可靠，只有写下来才可靠。

如果你能长期坚持这五件事，这个仓库后面即使继续增长，维护成本也不会失控。
